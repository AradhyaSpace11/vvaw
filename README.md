# VVA

VVA is the standalone workspace for the current robot visual imitation project.

The goal is to make a robot imitate a visual demonstration video, including videos generated by an image-to-video model. In the intended research workflow, an image of the robot's current scene is given to an I2V model with a task prompt, such as "pick up the ball and place it near the box." The generated video becomes the visual demonstration. VVA then grounds that generated motion in the real robot's own camera view and predicts robot actions that execute the same intent.

This means VVA is not only replaying recorded joint angles. It is meant to be a bridge:

```text
current robot camera view + visual future/demo video -> robot actions
```

For now, the project is focused on a single detected object. The detector already has a `Target` class for the object.

## Current Status

The older root-level scripts were previous iterations. The cleaned project lives inside this `vva/` folder and is intended to be portable: if this folder is copied out of the old root repository, it should still contain the data, YOLO assets, robot URDF, subsystem scripts, approach folders, and training/inference files needed for further development.

The active approaches are:

```text
v1/   older 3D-feature visual prompt approach
v2/   current intent-relative single-object inference approach
v3/   new delta-action experiment folder
```

The newest work is `v3/`. It uses the same 2D dataset as v2, but trains a better model target:

```text
intent-relative visual features
delta joint actions
separate arm and gripper prediction heads
```

## Data Story

The raw paired data lives in:

```text
data/
  demovideos/   reference task videos
  camview/      camera recordings while imitating each demo in simulation
  jointdata/    teleop joint commands corresponding to each camview
```

Naming is trial-based:

```text
data/demovideos/demovidN.mp4
data/camview/camviewN.mp4
data/jointdata/jdN.csv
```

Trials `1` through `9` are complete paired examples. `demovid10.mp4` exists as an extra visual demo, but it does not currently have matching `camview10.mp4` and `jd10.csv`, so it is not part of the supervised dataset.

The collection process was:

1. Watch a `demovidN.mp4`.
2. Use the physical teleop arm to make the simulated robot imitate that video.
3. Record the simulation camera view as `camviewN.mp4`.
4. Record the teleop joint targets as `jdN.csv`.
5. Extract visual point features from both the demo video and camview video.
6. Train a model to map visual prompt + current camera state to robot action.

The joint CSV format is:

```csv
timestamp,j0,j1,j2,j3,j4,j5,d0,d1,d2,d3,d4,d5
```

`j0` through `j5` are target joint positions. `d0` through `d5` are per-frame deltas.

## YOLO Points

The detector currently uses 7 classes:

```text
0 J0_Base
1 J1_Shoulder
2 J2_Elbow
3 J3_Wrist
4 J4_GripL
5 J5_GripR
6 Target
```

The v2 and v3 2D dataset stores 14 raw values per frame:

```text
J0 x,y
J1 x,y
J2 x,y
J3 x,y
J4 x,y
J5 x,y
Target x,y
```

For the full future task "pick object and place near goal," the detector will eventually need separate roles such as `Object` and `Goal`. For the current single-object version, one `Target` object class is enough.

## Teleop Controller

`teleop.ino` is the ESP32 sketch for the physical potentiometer teleop arm.

It reads:

```text
yaw       GPIO 36
shoulder  GPIO 39
elbow     GPIO 34
end link  GPIO 35
button    GPIO 12
```

On startup, it calibrates for 5 seconds. During calibration, hold the teleop arm at its zero pose. After calibration it streams:

```text
yaw,shoulder,elbow,endlink,button
```

The PC-side collection code expects this 5-value serial format at `115200` baud. The button is mapped to the gripper joints. Calibration lines start with `CAL` and are ignored by the Python reader.

GPIO `12` is an ESP32 boot-strapping pin. It can work, but if flashing or booting behaves strangely while the button is pressed, move the button to a safer GPIO such as `14`, `25`, `26`, or `27`.

## v2

`v2/` is the current single-object intent-relative approach.

```text
dataset:        v2/dataset_2d.npy
dataset maker:  v2/extract_points_2d.py
Colab trainer:  v2/colab_train_intent_single_cell.py
inference:      v2/infer.py
model:          v2/model2_vla_2d_intent.pth
```

The important change from the older v2 model is that raw 14D YOLO points are converted into intent-relative features before being sent to the model. These include arm shape relative to the base, gripper-to-target direction, gripper-to-target distance, and gripper width.

Current v2 inference is consolidated into one file:

```text
v2/infer.py
```

Older helper files like `infer2.py` and `intent_features.py` are obsolete. If an IDE still has those tabs open, they are stale tabs from before cleanup.

Run v2 inference:

```bash
cd ~/vva
source .venv/bin/activate
python v2/infer.py --demo 5 --phase-speed 0.75 --physics-steps 10
```

Use demos `1` through `9` for trained behavior. `demovid10.mp4` is only an unpaired extra demo for now.

## v3

`v3/` is the new model-improvement experiment.

```text
dataset:       v3/dataset_2d.npy
Colab trainer: v3/train_v3_delta_intent_single_cell.ipynb
output model:  v3/model3_delta_intent.pth
```

v3 keeps the same visual idea as v2, but changes the action target:

```text
v2 predicts absolute joint targets
v3 predicts delta joint actions
```

v3 also uses separate output heads:

```text
arm head      -> j0,j1,j2,j3 deltas
gripper head  -> j4,j5 deltas
```

This should help reduce freezing/averaging and make gripper behavior less likely to be ignored by the loss.

Train v3 in Colab:

1. Open `v3/train_v3_delta_intent_single_cell.ipynb`.
2. Use a T4 GPU runtime.
3. Run the single cell.
4. Upload `v3/dataset_2d.npy` if Colab asks.
5. Download `model3_delta_intent.pth`.
6. Put the downloaded model in `~/vva/v3/`.

There is not yet a v3 inference script. The v2 `infer.py` should not be used directly with v3 because v3 checkpoints store `delta_mean` and `delta_std` and predict action deltas instead of absolute targets. The next step after training v3 is creating `v3/infer.py`.

## Setup

From inside this project folder (`C:\vvaw` on this Windows machine):

### Windows / VVAW

PowerShell:

```powershell
.\setup_windows.ps1
.\run_smoke_windows.ps1
.\run_v2_windows.ps1 -Demo 5
```

For NVIDIA GPU acceleration, install the CUDA PyTorch wheels after the normal setup:

```powershell
.\setup_windows.ps1 -Cuda
.\check_gpu_windows.ps1
```

If PowerShell blocks local scripts for the current terminal session:

```powershell
powershell -ExecutionPolicy Bypass -File .\setup_windows.ps1
```

The Windows setup uses `pybullet-arm64`, a drop-in `pybullet` package with Windows wheels, so it does not need Microsoft C++ Build Tools.

For a quick non-GUI check:

```powershell
.\run_v2_windows.ps1 -Demo 1 -Headless -MaxSteps 1 -PhysicsSteps 1 -PhaseSpeed 1.0
```

### macOS / Linux

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

If using the already-created standalone copy at `~/vva`, activate it with:

```bash
cd ~/vva
source .venv/bin/activate
```

## Development Rule For New Approaches

For a new approach, make a new folder:

```text
v4/
  dataset or copied dataset
  train.py or train.ipynb
  infer.py
  model file
```

Keep generated `.pth` files inside the same approach folder. That way each approach stays self-contained and can be tested without depending on older root-level scripts.
